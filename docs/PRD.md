# NexusGraph — 產品需求文件 (PRD)

## 1. 產品願景

NexusGraph 是一個雲端同步的 Git Repo 知識圖譜平台，專為開發者設計，自動解析程式碼之間的呼叫關係、模組依賴與資料流向，以互動式知識圖譜呈現。目標使用者為需要快速理解陌生程式碼架構的開發者，或希望提升團隊協作效率的工程團隊。核心價值在於將「知識圖譜」作為主要 UI 介面，而非傳統的搜尋框，讓開發者可以直接點擊圖譜節點，快速理解陌生程式碼的架構邏輯，將 onboarding 時間從數天縮短到數小時。

## 2. User Story

### Story 1：個人開發者快速理解陌生 Repo
**身分**：作為一名個人開發者，我希望能夠上傳一個 GitHub Repo URL 並在數分鐘內獲得該 Repo 的互動式知識圖譜，以便我能快速理解陌生專案的架構邏輯，而不需要手動追蹤數百個檔案之間的關係。

**輸入**：GitHub Repo URL（例如 https://github.com/facebook/react）
**輸出**：一張互動式知識圖譜，節點代表模組/檔案，邊代表呼叫關係或依賴關係
**成功標準**：
- 圖譜在 2 分鐘內生成完畢
- 使用者點擊任一節點，能看到該模組的函式清單與被呼叫位置
- 圖谱支援縮放、拖曳、節點搜尋

### Story 2：團隊成員協作註釋與文件生成
**身分**：作為一個工程團隊的成員，我希望能在知識圖譜上對特定模組或函式新增註釋、標註風險等級，並能一鍵生成模組說明文件，以便團隊新成員能自助理解程式碼並传承經驗。

**輸入**：在圖譜上選取節點 → 新增文字註釋或風險標籤
**輸出**：可匯出的 Markdown 文件，包含模組說明、呼叫關係、團隊註釋
**成功標準**：
- 支援多人同時對同一節點新增不同註釋
- 註釋內容可一鍵匯出成 Markdown
- 文件包含該模組的 Jina embeddings 自動生成的說明摘要

### Story 3：團隊管理者檢視 Repo 健康度與依賴關係
**身分**：作為一個工程團隊的 Tech Lead，我希望能看到團隊成員上傳的所有 Repo 的依賴關係概覽圖，以便我能識別重複引入的 Library、發現潛在的循環依賴、或規劃重構範圍。

**輸入**：團隊 dashboard，列出所有已分析的 Repo
**輸出**：
- 各 Repo 的依賴圖
- 跨 Repo 的共享模組識別
- 循環依賴警告

**成功標準**：
- Dashboard 能在單一頁面呈現團隊所有 Repo 的狀態
- 循環依賴被自動標記為紅色節點
- 可追蹤特定 Library 被哪些 Repo 使用

## 3. 功能分級

### P0（MVP，必須在 Prototype 完成）

| ID | 功能名稱 | 說明 |
|----|----------|------|
| P0-1 | GitHub Repo 輸入與 Clone | 使用者輸入 GitHub URL，系統自動 Clone 並解析 |
| P0-2 | 呼叫關係圖谱解析 | 解析程式碼之間的 function call / import / require 關係 |
| P0-3 | 互動式圖谱展示 | 使用 D3.js 或 Cytoscape.js 渲染可縮放、可拖曳的節點圖 |
| P0-4 | 節點點擊詳情 | 點擊節點顯示該檔案/模組的內容摘要、函式清單、被呼叫位置 |
| P0-5 | 使用者登入/註冊 | 支援 Email/Password 註冊與登入（Freemium 基礎） |

### P1（Beta 版本新增）

| ID | 功能名稱 | 說明 |
|----|----------|------|
| P1-1 | 團隊協作功能 | 建立團隊、邀請成員、檢視團隊所有 Repo |
| P1-2 | 節點註釋系統 | 對圖谱節點新增文字註釋、風險標籤 |
| P1-3 | Markdown 文件匯出 | 將圖谱 + 註釋一鍵匯出成 Markdown 文件 |
| P1-4 | Jina Embeddings 自動說明 | 對每個模組自動生成簡短說明（调用 Jina API） |
| P1-5 | Repo 健康度儀表板 | 顯示檔案大小、依賴數量、循環依賴警告 |

### P2（企業版功能）

| ID | 功能名稱 | 說明 |
|----|----------|------|
| P2-1 | SSO 單一登入 | 支援 Google/GitHub OAuth SSO |
| P2-2 | 私有部署選項 | Docker 部署到企業內部伺服器 |
| P2-3 | API 存取 | 支援外部系統透過 REST API 查詢圖谱資料 |
| P2-4 | 存取控制列表 (ACL) | 精細控制誰可以看/編輯哪些 Repo |

## 4. 非功能需求

### 效能
- Repo 圖谱解析時間：< 2 分鐘（對於 1000 個檔案以內的 Repo）
- 圖谱渲染幀率：>= 30 FPS（拖曳/縮放時）
- API 回應時間：P95 < 500ms

### 可用性
- 系統 uptime：>= 99.5%（每月 downtime < 3.65 小時）
- 錯誤回復：系統崩潰後能在 5 分鐘內重啟並恢復資料

### 擴展性
- 支援同時分析至少 50 個 Repo（團隊版）
- 資料庫支援水平分片，未來可擴展到百萬節點規模

### 安全性
- 所有 API 使用 HTTPS
- 使用者密碼使用 bcrypt 雜湊（cost factor >= 12）
- JWT token 有效期 <= 24 小時，支援 refresh token

## 5. 技術選型

### 前端
- **Framework**：Next.js 14 (App Router) + TypeScript
- **圖谱渲染**：Cytoscape.js（支援大型圖谱的 WebGL 加速）
- **狀態管理**：Zustand
- **選擇理由**：Next.js 提供 SSR / ISR 能力，適合 SEO 導向的落地頁；Cytoscape.js 在 GitHub 有 9k+ stars，生態成熟，文件完整，相較 D3.js 更適合知識圖谱互動操作。

### 後端
- **Framework**：Node.js + Express + TypeScript
- **Graph 解析**：Python（用 AST 解析程式碼關係）+ FastAPI 作為微服務
- **選擇理由**：Node.js 配合 Express 生態完整，Python 的 `ast` 模組和 `tree-sitter` 能精確解析多語言程式碼關係，將解析服務剝離成 Python FastAPI 微服務，可獨立擴展。

### 資料庫
- **主要儲存**：PostgreSQL + PostGIS（儲存 Repo 資料與使用者資料）
- **圖谱儲存**：Neo4j（原生支援圖結構查詢，Cypher 語法適合關係遍歷）
- **快取**：Redis（圖谱節點快取、API 結果快取）
- **選擇理由**：Neo4j 的 Cypher 查詢比 SQL 的 JOIN 在多跳關係查詢上快 10-100 倍，非常適合知識圖谱的「找到所有被 A 呼叫但不被 B 呼叫的模組」這類查詢。

### 部署工具
- **容器化**：Docker + Docker Compose
- **雲端平台**：Vercel（前端的部署）+ Railway（後端 + 資料庫）
- **CI/CD**：GitHub Actions
- **選擇理由**：Vercel 對 Next.js 有原生支援，冷啟動速度快；Railway 支援直接部署 PostgreSQL + Neo4j，省去管理基礎設施的麻煩。

## 7. 成功指標

### KPI 1：圖谱生成速度
- **指標**：從使用者粘貼 GitHub URL 到圖谱完整渲染的時間
- **目標**：MVP 版本 < 2 分鐘（1000 檔案以內）
- **測量方式**：在 Beta 使用者同意下，記錄每次分析的完成時間，取 P50 與 P95
- **達標標準**：P50 <= 90 秒，P95 <= 120 秒

### KPI 2：用戶留存率
- **指標**：首次使用後 30 天內回訪並分析第二個 Repo 的使用者比例
- **目標**：> 30%
- **測量方式**：比較每月新增使用者數 vs. 30 天後仍在活躍的使用者數
- **達標標準**：30 天留存率 >= 30%，60 天留存率 >= 20%

### KPI 3：錯誤率
- **指標**：API 請求失敗率（P95 回應時間超標也算失敗）
- **目標**：< 1%
- **測量方式**：在 API Gateway 層记录 5xx 錯誤與超時
- **達標標準**：錯誤率 < 1%，系統 uptime >= 99.5%

### KPI 4：使用量成長
- **指標**：每週活躍 Repo 分析數
- **目標**：上線後 12 週內，週活躍 Repo 數從 100 成長到 1000（10x）
- **測量方式**：每週統計有多少個不同的 Repo URL 被分析（去重）
- **達標標準**：第 12 週週活躍 Repo >= 1000
